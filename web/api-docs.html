API Reference

PrismRAG API

PrismRAG is an API-first platform. Everything available in the dashboard is also available via REST — workspace creation, document ingestion, semantic search, mapping management, and Deliberation sessions.

The full machine-readable specification is available at /docs (Swagger UI) and /openapi.json.

💡
Quickstart: Register → get a JWT → create a workspace → ingest documents → search. See the Playground to try it instantly with no code.

Base URL

https://api.prismrag.io/api/v1

All endpoints below are relative to this base. The API is versioned — v1 is stable and will not receive breaking changes without a version bump.

Authentication

All protected endpoints require a Bearer JWT in the Authorization header.

Register

POST/auth/register
Create a new account. Returns a JWT on success.
# Request
{
  "email":     "you@company.com",
  "password":  "YourPassword123!",
  "full_name": "Ada Lovelace"        // optional
}

# Response 201
{
  "token": "eyJhbGci...",
  "email": "you@company.com"
}

Login

POST/auth/login
Exchange email + password for a JWT.
curl -X POST https://api.prismrag.io/api/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"you@company.com","password":"YourPassword123!"}'

Using the token

curl https://api.prismrag.io/api/v1/auth/me \
  -H "Authorization: Bearer eyJhbGci..."
⚠️
Token lifetime: JWTs expire after 30 days. Re-authenticate using POST /auth/login to get a fresh token. Enterprise plans can request service-account tokens with configurable expiry.

Errors & rate limits

CodeMeaning
400Bad request — check your JSON body
401Missing or expired token
402Plan limit reached — upgrade or wait for reset
403Forbidden — you don't own this resource
404Resource not found
422Validation error — see detail array
429Rate limit exceeded — back off and retry
500Server error — contact support if persistent

Rate limits are enforced per API key per minute. Retry with exponential backoff on 429. The Retry-After header gives the reset time in seconds.

Workspaces

A workspace (tenant) is an isolated knowledge graph. All ingested content, mappings, communities, and search results are scoped to a workspace.

POST/prismrag/tenants
Create a new workspace.
{ "name": "My Knowledge Base" }

# Response
{ "tenant_id": "wsp_01abc...", "name": "My Knowledge Base" }
GET/prismrag/tenants
List all workspaces owned by the authenticated user.
DELETE/prismrag/tenants/{tenant_id}
Permanently delete a workspace and all its data.

Mappings

A mapping defines your knowledge graph structure — which categories exist and which words belong to each. PrismRAG uses mappings to project ingested content into your semantic model.

POST/prismrag/mappings
Create or update the mapping version for a workspace. Pass strategy: "rules" for deterministic projection or "mlp" to train a personal neural model.
{
  "tenant_id": "wsp_01abc...",
  "strategy":  "rules",          // "rules" | "mlp"
  "categories": [
    {
      "category_slug": "risk",
      "label": "Risk & Compliance",
      "words": [
        { "word": "volatility", "weight": 1.0 },
        { "word": "exposure",   "weight": 0.8 }
      ]
    }
  ]
}
💡
You don't need to list every word. Tier-1 rules cover exact matches. Tier-2 (MLP) generalises to vocabulary not in your list — train it after you've defined at least 10 words per category.
GET/prismrag/mappings?tenant_id={id}
Retrieve the current mapping version for a workspace.

Ingest jobs

Ingest jobs embed, map, and index your content. Jobs are asynchronous — submit the job, then poll until status: "completed".

POST/prismrag/jobs
Submit an ingest job. Supports inline records, Azure Blob URLs, or file uploads.
# Inline records
{
  "tenant_id":   "wsp_01abc...",
  "strategy":    "rules",
  "source_type": "inline",
  "records": [
    { "id": "doc-1", "text": "Volatility in Q3 was driven by..." }
  ]
}

# Azure Blob (large files)
{
  "tenant_id":   "wsp_01abc...",
  "strategy":    "mlp",
  "source_type": "blob",
  "blob_url":    "https://storage.azure.com/..."
}
GET/prismrag/jobs/{job_id}
Poll job status. Returns status, progress_pct, and records_written when complete.
# Response (in progress)
{ "status": "running", "progress_pct": 60 }

# Response (complete)
{ "status": "completed", "records_written": 142, "duration_s": 14.2 }

Search

Semantic search over your indexed content. Queries are routed through your community graph and re-ranked by your personal model when available.

POST/prismrag/search
Run a semantic search query against a workspace.
{
  "query":     "What are the main risk factors in Q3?",
  "tenant_id": "wsp_01abc...",
  "top_k":     10,
  "strategy":  "rules"           // optional — defaults to workspace strategy
}

# Response
{
  "results": [
    {
      "chunk_id":   "chk_...",
      "text":       "Volatility in Q3 was driven by...",
      "score":      0.94,
      "categories": ["risk"]
    }
  ],
  "latency_ms": 38
}

Communities

Communities are automatically detected semantic clusters in your knowledge graph. Each has a name, a set of member words, and a centroid vector used for query routing.

GET/prismrag/communities?tenant_id={id}
List all communities detected in a workspace after the last ingest job.
[
  {
    "community_id": "com_...",
    "label":        "Risk & Compliance",
    "word_count":   48,
    "chunk_count":  312
  }
]

Bridge vectors

Bridge vectors create traversable links between two knowledge communities — enabling cross-domain retrieval without retraining.

POST/prismrag/bridges
Create a semantic bridge between two communities.
{
  "tenant_id":      "wsp_01abc...",
  "community_id_a": "com_risk_...",
  "community_id_b": "com_legal_...",
  "label":          "Risk-Legal overlap"   // optional
}

Deliberation — Create session

A Deliberation session represents one complex question run through the multi-domain expert pipeline. Sessions are async — create, run, then poll.

POST/deliberation/sessions
Create a new deliberation session.
{
  "query":     "Should we proceed with the acquisition given current market conditions?",
  "tenant_id": "wsp_01abc..."    // optional — grounds domains in your KB
}

# Response
{ "session_id": "dlb_...", "status": "created" }

Deliberation — Run pipeline

POST/deliberation/sessions/{id}/run
Start the three-phase deliberation pipeline. Returns immediately; poll for results.
⏱
Typical latency: 15–45 seconds end-to-end depending on domain complexity. Poll every 3–5 seconds.

Deliberation — Poll & retrieve

GET/deliberation/sessions/{id}
Get session status and results. When status: "completed" the full synthesis, domain summaries, agreements, and conflicts are included.
{
  "status": "completed",
  "domains_discovered": ["Finance", "Antitrust", "HR", "Operations"],
  "domain_summaries": { "Finance": "DCF supports valuation at..." },
  "agreements": ["All domains agree integration timeline is 18 months"],
  "conflicts":  ["Finance: proceed · Antitrust: 40% block probability"],
  "synthesis":  "Based on independent expert analysis..."
}

Deliberation — Follow-up

POST/deliberation/sessions/{id}/followup
Ask a follow-up question on a completed session. Does not consume your monthly deliberation quota.
{ "question": "What specifically did the Antitrust domain flag?" }

Profile

GET/auth/me
Get the authenticated user's profile, plan, and tenant ID.

Usage

GET/dashboard/usage
Current period usage against plan limits — searches, deliberations, chunks, and overage.
{
  "usage":     { "searches": 1240, "deliberations": 18 },
  "limits":    { "searches": 50000, "deliberations": 500 },
  "remaining": { "searches": 48760, "deliberations": 482 }
}

Billing

GET/billing/plans
List available plans with pricing and limits.
POST/billing/checkout
Create a Stripe Checkout session to upgrade the authenticated user's plan.
POST/billing/portal
Open a Stripe Customer Portal session for managing subscriptions and invoices.

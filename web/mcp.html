MCP Integration

Connect AI agents to PrismRAG

PrismRAG ships a built-in Model Context Protocol (MCP) server that exposes your knowledge graph and Deliberation engine as tools any MCP-compatible AI agent can call — Claude, ChatGPT, custom agents built on the Claude API, and more.

With MCP, your agent can search your domain-specific knowledge base, launch multi-domain deliberation sessions, and manage workspaces — all from within the model's tool-use interface, with no custom integration code.

💡
No SDK needed. PrismRAG's MCP server works out of the box with any MCP client. For Claude Code or Claude Desktop, copy the config snippet below. For custom agents, use the HTTP transport endpoint.

The MCP server exposes all PrismRAG and Deliberation functionality as named tools. Agents call them the same way they call any other tool — no special handling required.

search list_workspaces list_communities submit_job get_job_status create_bridge deliberate get_deliberation_session deliberation_followup

Quickstart — Claude Desktop / Claude Code

Add PrismRAG to your claude_desktop_config.json (Claude Desktop) or .claude/mcp.json (Claude Code). Replace YOUR_TOKEN with your API token from the dashboard.

Option A — HTTP transport (recommended)

{
  "mcpServers": {
    "prismrag": {
      "transport": "http",
      "url":       "https://api.prismrag.io/mcp",
      "headers": {
        "Authorization": "Bearer YOUR_TOKEN"
      }
    }
  }
}

Option B — stdio transport (local / self-hosted)

{
  "mcpServers": {
    "prismrag": {
      "command": "uvx",
      "args": ["prismrag-mcp"],
      "env": {
        "PRISMRAG_API_TOKEN": "YOUR_TOKEN",
        "PRISMRAG_API_URL":   "https://api.prismrag.io"
      }
    }
  }
}
✅
Verify it's working: ask your agent "List my PrismRAG workspaces" — it should call list_workspaces and return your workspace names.

Transport options

The HTTP transport connects your agent directly to the PrismRAG cloud API. No local installation required. Authentication is via the Authorization: Bearer header.

# MCP endpoint
https://api.prismrag.io/mcp

# Headers required on every request
Authorization: Bearer YOUR_TOKEN
Content-Type: application/json

The stdio transport runs a local process that proxies MCP calls to the PrismRAG API. Useful for self-hosted deployments or air-gapped environments.

# Install the MCP server package
pip install prismrag-mcp

# Run directly (for testing)
PRISMRAG_API_TOKEN=YOUR_TOKEN \
PRISMRAG_API_URL=https://api.prismrag.io \
python -m prismrag_mcp
⚠️
Self-hosted: point PRISMRAG_API_URL at your own instance for on-premises deployments. Enterprise plan required.

Use the PrismRAG Python SDK to call MCP tools programmatically from your own agent or workflow.

pip install prismrag
from prismrag import PrismRAGClient

client = PrismRAGClient(token="YOUR_TOKEN")

# Search your knowledge base
results = client.search(
    query="What are the main risk factors in Q3?",
    tenant_id="wsp_01abc...",
    top_k=10
)

# Run a deliberation
session = client.deliberate(
    query="Should we proceed with the acquisition?",
    tenant_id="wsp_01abc..."
)
print(session.synthesis)

Tool: search

search
PrismRAG
Semantic search over a workspace's indexed content. Returns the top-K most relevant chunks with scores and category tags. Use this as the primary retrieval tool — pass results as context to the model for grounded answers.
ParameterTypeDescription
query * string Natural language search query
tenant_id * string Workspace ID to search within
top_k opt integer Number of results to return (default: 10, max: 50)
strategy opt string "rules" or "mlp" — defaults to workspace strategy 
# Example agent call (Claude tool-use format)
{
  "tool": "search",
  "input": {
    "query":     "Q3 earnings risk exposure",
    "tenant_id": "wsp_01abc...",
    "top_k":     8
  }
}

# Returns
{
  "results": [
    { "chunk_id": "chk_...", "text": "...", "score": 0.94, "categories": ["risk"] }
  ]
}

Tool: list_workspaces

list_workspaces
PrismRAG
List all workspaces owned by the authenticated user. Returns workspace IDs, names, and creation dates. Call this first if you don't know the tenant_id.
# No parameters required
{ "tool": "list_workspaces", "input": {} }

Tool: list_communities

list_communities
PrismRAG
List the semantic communities detected in a workspace. Useful for understanding the knowledge graph structure before searching, or for identifying which communities to bridge.
ParameterTypeDescription
tenant_id * string Workspace to inspect

Tool: submit_job

submit_job
PrismRAG
Submit an ingest job to embed and index new content into a workspace. Pass records as an array of {id, text} objects. Returns a job_id — use get_job_status to poll for completion.
ParameterTypeDescription
tenant_id * string Target workspace
records * array Array of {id: string, text: string} objects
strategy opt string "rules" (default) or "mlp"

Tool: get_job_status

get_job_status
PrismRAG
Poll an ingest job for status and progress. Returns status (queued / running / completed / failed), progress_pct, and records_written when complete.
ParameterTypeDescription
job_id * string Job ID returned by submit_job

Tool: create_bridge

create_bridge
PrismRAG
Create a semantic bridge between two knowledge communities. This enables cross-domain retrieval — queries near one community can traverse the bridge to retrieve content from the other, without retraining the model.
ParameterTypeDescription
tenant_id * string Workspace containing the communities
community_id_a * string First community ID
community_id_b * string Second community ID
label opt string Human-readable name for this bridge

Tool: deliberate

deliberate
Deliberation
Submit a complex question for multi-domain expert deliberation. This is an async operation — the tool returns a session_id immediately. Use get_deliberation_session to poll for the completed synthesis, domain summaries, agreements, and conflicts.
ParameterTypeDescription
query * string The complex question to deliberate on
tenant_id opt string Workspace ID — grounds each domain expert in your KB 
# Typical agent pattern
1. Call deliberate → get session_id
2. Poll get_deliberation_session every ~5s until status = "completed"
3. Present synthesis + conflicts to the user
4. Use deliberation_followup for follow-up questions (no quota charge)

Tool: get_deliberation_session

get_deliberation_session
Deliberation
Retrieve the status and results of a deliberation session. When status: "completed", the response includes: synthesis, domain_summaries, agreements, conflicts, and domains_discovered.
ParameterTypeDescription
session_id * string Session ID from the deliberate call

Tool: deliberation_followup

deliberation_followup
Deliberation
Ask a follow-up question on a completed deliberation session. The answer is synthesised from the existing domain results — no new deliberation pipeline is run, so this does not consume your monthly quota. Ideal for drilling into specific domain conflicts or asking for clarification.
ParameterTypeDescription
session_id * string Completed deliberation session ID
question * string Follow-up question

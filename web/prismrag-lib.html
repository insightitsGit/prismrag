How it works

Three steps. Zero infrastructure change.

PrismRAG Patch wraps your existing database client. Your data never leaves your environment.

Step 01

Define your mapping

Write a category + rules dict in Python — exactly like you'd write a taxonomy. "fever" → symptoms. "antibiotics" → treatment. Your domain expertise, your rules.

Tier 1 · Deterministic
Step 02

Wrap your DB client

Pass your existing connection or collection to a PrismRAG adapter. No schema changes. No data migration. One import and two lines of code.

Drop-in · Same API
Step 03

Every vector is re-mapped

On insert: each embedding is blended with your category signal before storage. On search: the query vector is re-mapped identically so it stays in the same space. Same-category chunks cluster together — wrong chunks no longer surface.

Auditable · Reproducible

Quickstart

From zero to grounded retrieval in 5 minutes.

quickstart_pgvector.py 
# 1. Install
#    pip install "prismrag-patch[graph,pgvector]"

from prismrag_patch import PrismRAG, PrismRAGPatch
from prismrag_patch.adapters.pgvector import PgvectorAdapter

# 2. Full client (ingest + graph RAG) or adapter-only remap
rag = PrismRAG(mapping={
        "categories": [
            {"slug": "symptoms",  "label": "Symptoms & Signs"},
            {"slug": "treatment", "label": "Treatment & Therapy"},
            {"slug": "diagnosis", "label": "Diagnosis"},
        ],
        "rules": [
            {"word": "fever",        "category_slug": "symptoms"},
            {"word": "headache",     "category_slug": "symptoms"},
            {"word": "antibiotics",  "category_slug": "treatment"},
            {"word": "hypertension", "category_slug": "diagnosis"},
        ],
    },
    tenant_id="demo",
)

patch = PrismRAGPatch(mapping=rag.mapping)

# 3. Wrap your existing connection — no schema changes
adapter = PgvectorAdapter(patch, dsn="postgresql://user:pass@localhost/mydb")

# 4. Insert with automatic re-mapping
my_vector = embed("Patient presents with fever and headache")   # your embedding fn
adapter.insert(text="Patient presents with fever and headache", vector=my_vector)

# 5. Search — query is re-mapped identically, so similarity is preserved
results = adapter.search("fever treatment", query_vector=q_vec, top_k=5)
# → returns verified 'symptoms' and 'treatment' chunks — not river-bank noise

FAQ

Common questions

Does my data ever leave my environment?
No. All re-mapping and graph RAG run in your Python process. Embeddings, queries, and documents never go to a PrismRAG hosted API. Use MemoryStore locally or PostgresStore on your database.
Do I need a license key?
No. The library is Apache-2.0 on PyPI with no license validation or phone-home.
Do I need to change my database schema?
For adapters: no — vectors are remapped in memory before INSERT. For PostgresStore: apply the prismrag.* schema once (prismrag/schema.sql).
What about existing embeddings already in my database?
Backfill by fetching them, applying patch.remap_vector(vector, text=...), and updating. Or re-ingest through PrismRAG.ingest() for the full graph pipeline.
Can I use my own embedding model?
Yes — pass embed_fn to PrismRAG or embed externally for adapters. Any dimension works.
Full client vs adapter-only?
PrismRAG gives ingest, graph RAG, communities, bridges, and append. Adapters add Tier-1 remap to your existing vector DB — use PostgresStore for graph features on Postgres.
